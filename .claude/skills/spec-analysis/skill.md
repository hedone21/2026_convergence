# Spec Analysis — 요구사항 → 정형 스펙 변환

요구사항 문서를 분석하여 개별 ID가 부여된 정형 스펙 문서를 생성한다. 이 스펙은 아키텍처, 구현, 테스트의 공통 기준이 된다.

## 핵심 원칙: 스펙의 추상성

스펙은 **기준 문서**이다. 구현이 바뀌어도 스펙은 변하지 않아야 한다.

**스펙에 포함하는 것** (What — 무엇을):
- 시스템이 달성해야 하는 동작과 결과
- 구현 기술에 독립적인 성공/실패 조건
- 예외 상황과 대응 방침
- 필요시 의사코드(pseudocode) 수준의 로직

**스펙에 포함하지 않는 것** (How — 어떻게):
- 우선순위, 할당 → `docs/todo.md`에서 관리
- 구체적 클래스명 (예: `HazardManager`, `XRInterface`) → 아키텍처 문서의 역할
- 구체적 API 호출 (예: `xr_interface.is_initialized()`) → 아키텍처/코드의 역할
- 프레임워크 특정 코드 → 아키텍처/코드의 역할

**좋은 성공 조건 vs 나쁜 성공 조건:**

| 나쁜 예 (구현 종속) | 좋은 예 (추상적) |
|-------------------|----------------|
| `xr_interface.is_initialized()`가 `true` 반환 | VR 환경이 초기화되어 스테레오 렌더링이 활성화된다 |
| `HazardManager.get_hazard_count()` == 3 | 시나리오에 배치된 모든 위험 요소가 씬에 로드된다 |
| `Area3D`의 `body_entered` 시그널이 발생 | 플레이어가 위험 요소 감지 범위에 진입하면 탐지 이벤트가 발생한다 |
| CSV 파일에 `timestamp` 컬럼 존재 | 모든 이벤트 기록에 시간 정보가 포함된다 |

## 변환 절차

### Step 1: 요구사항 분석
`docs/requirements.md`를 읽고 기능 요구사항을 식별한다:
- 각 절에서 구현해야 할 기능을 추출
- 암시적 요구사항도 식별 (예: "시간 제한 있음" → 타이머 기능 필요)

### Step 2: 스펙 단위 분해
추출한 기능을 **하나의 검증 가능한 단위**로 분해한다:
- 한 스펙 = 한 가지 독립적으로 테스트 가능한 기능
- 너무 크면 분할: "VR 환경 전체" → "VR 초기화" + "VR 렌더링" + "컨트롤러 입력"
- 너무 작으면 통합: "버튼 색상 변경" 수준은 스펙이 아니라 구현 디테일

### Step 3: Spec ID 부여

```
SPEC-{영역}-{순번}

영역 코드:
  VR  — VR 관련
  ENV — 3D 환경
  HAZ — 위험 요소
  SCN — 시나리오 관리
  DAT — 데이터 수집
  SES — 세션 관리
  UI  — UI
  INP — 입력/인터랙션
```

### Step 4: 성공/실패 조건 정의

**성공 조건**: 구현 독립적이면서 측정 가능한 동작 결과
**실패 조건**: 명확히 실패로 판정해야 하는 상태
**예외 처리**: 예상 가능한 비정상 상황과 시스템의 대응
**대안 동작**: 주 기능이 불가할 때의 폴백. 해당 없으면 "없음"

### Step 5: 실행 계획 작성

`docs/todo.md`에 별도로 작성한다:
- 각 Spec ID에 대해 우선순위 (P0/P1/P2), 할당 (senior-dev/dev), 의존성
- 구현 순서 권장안
- 진행 상태 추적

### Step 6: 출력 문서 생성

#### 파일 1: `docs/specs.md`

```markdown
# 정형 스펙 문서

## 메타데이터
- 요구사항 문서: docs/requirements.md
- 생성일: {날짜}
- 총 스펙 수: {N}개

## 요구사항 커버리지 매트릭스
| 요구사항 절 | 커버 Spec ID |
|-----------|-------------|
| ... | ... |

## 스펙 요약
| Spec ID | 제목 |
|---------|------|
| SPEC-VR-001 | ... |

## 스펙 상세

### SPEC-VR-001: {제목}
- **출처**: 요구사항 {절}
- **설명**: {추상적 기능 설명}
- **성공 조건**:
  - {구현 독립적 조건}
- **실패 조건**:
  - {구현 독립적 실패 기준}
- **예외 처리**:
  - {예외}: {대응}
- **대안 동작**: {폴백 또는 "없음"}
- **비고**: {참고}
```

#### 파일 2: `docs/todo.md`

```markdown
# 실행 계획 (Todo)

## 요약
- 총 스펙: {N}개
- P0(MVP): {n}개 / P1: {n}개 / P2: {n}개

## 태스크 목록
| Spec ID | 제목 | 우선순위 | 할당 | 의존성 | 상태 |
|---------|------|---------|------|--------|------|
| SPEC-VR-001 | ... | P0 | senior-dev | 없음 | pending |

## 구현 순서 권장안
{Phase별 순서}

## 의존성 다이어그램
{텍스트 다이어그램}
```

## 품질 체크리스트
- [ ] 모든 요구사항이 최소 1개 이상의 Spec ID로 커버됨
- [ ] 각 스펙의 성공 조건이 구현 독립적이고 테스트로 검증 가능
- [ ] 스펙에 클래스명, 구체적 API, 프레임워크 코드가 없음
- [ ] 우선순위와 할당이 specs.md가 아닌 todo.md에 있음
- [ ] Spec ID 체계가 일관됨 (영역코드-순번)
