### SPEC-VR-001: VR 환경 초기화 및 세션 시작

- **출처**: 요구사항 8절 (기술 스택 — VR HMD, OpenXR 호환)
- **설명**: VR 런타임을 초기화하고 HMD 디바이스에서 VR 세션을 시작한다. 스테레오 렌더링이 활성화되며, 원점(플레이어 위치 기준점), 머리 추적 카메라, 좌/우 컨트롤러가 준비된다.
- **성공 조건**:
  - VR 환경이 초기화되어 스테레오 렌더링이 활성화된다
  - 원점(origin), 머리 추적 카메라 1개, 좌/우 컨트롤러 각 1개가 인식된다
  - 초기화 후 첫 프레임이 3초 이내에 렌더링된다
- **실패 조건**:
  - VR 환경 초기화에 실패하여 스테레오 렌더링이 활성화되지 않는다
  - 초기화 후 5초 내에 첫 프레임이 렌더링되지 않는다
  - VR 세션 시작 시 런타임 에러가 발생한다
- **예외 처리**:
  - VR 런타임이 없는 환경 -> 에러 메시지를 로그에 기록하고 SPEC-VR-002 데스크톱 모드로 전환
  - 지원 대상 외 HMD 연결 -> OpenXR 호환 기기이면 정상 진행, 비호환이면 에러 로그 출력 후 데스크톱 모드 전환
- **대안 동작**: SPEC-VR-002 (데스크톱 모드 폴백)
- **비고**: 없음

### SPEC-VR-002: 데스크톱 모드 폴백

- **출처**: 요구사항 8절 (암시적 — 개발/테스트 편의를 위한 비VR 실행 필요)
- **설명**: VR 초기화 실패 시 또는 개발 편의를 위해 데스크톱(비VR) 모드로 실행할 수 있는 폴백을 제공한다. 키보드+마우스로 이동 및 마킹이 가능하다.
- **성공 조건**:
  - VR 초기화 실패 시 자동으로 데스크톱 모드로 전환되고, 전환 사유가 로그로 출력된다
  - 데스크톱 모드에서 키보드로 이동이 가능하다
  - 데스크톱 모드에서 마우스 클릭으로 마킹이 가능하다
  - 데스크톱 모드에서도 데이터 수집(SPEC-DAT-001)이 동일하게 동작한다
- **실패 조건**:
  - VR 초기화 실패 후 프로그램이 크래시되거나 정상 실행되지 않는다
  - 데스크톱 모드에서 이동 또는 마킹이 불가능하다
- **예외 처리**:
  - 데스크톱 모드도 실행 불가 -> 에러 메시지 표시 후 안전 종료
- **대안 동작**: 없음 (최종 폴백)
- **비고**: 커맨드라인 인자로 강제 데스크톱 모드 실행도 지원한다. VR 기기 없이도 테스트할 수 있어야 한다.

### SPEC-ENV-001: 건물 골조 현장 3D 환경

- **출처**: 요구사항 2.1절 (건물 골조 공사 현장, 철근콘크리트 구조물), 9절 MVP
- **설명**: 철근콘크리트 건물 골조 공사 현장을 3D로 구현한다. 기둥, 보, 슬래브, 벽체 등 기본 구조물과 바닥면으로 구성된, 걸어다닐 수 있는 환경을 제공한다.
- **성공 조건**:
  - 씬에 기둥 최소 4개, 보 최소 4개, 슬래브 최소 1개가 존재한다
  - 플레이어가 걸어다닐 수 있는 바닥면에 충돌 판정이 존재한다
  - 플레이어가 구조물 내부에서 이동 시 벽/기둥을 통과하지 않는다 (충돌 판정 정상 동작)
  - VR 디바이스에서 60fps 이상의 프레임레이트를 유지한다
- **실패 조건**:
  - 씬 로드 시 에러가 발생한다
  - 플레이어가 바닥을 뚫고 떨어진다
  - VR 디바이스에서 프레임레이트가 60fps 미만이다
- **예외 처리**:
  - 저사양 기기에서 프레임 드롭 -> LOD(Level of Detail) 또는 구조물 수 축소 설정 제공
- **대안 동작**: 없음
- **비고**: 에셋이 없으므로 프로시저럴 지오메트리로 구성. 추후 실제 에셋으로 교체 가능한 구조로 설계.

### SPEC-ENV-002: 크랙 절차적 생성 시스템

- **출처**: 요구사항 2.4절 (절차적 생성), 9절 MVP (크랙 3개 배치)
- **설명**: 콘크리트 표면에 크랙(균열)을 절차적으로 생성하는 시스템을 구현한다. 크랙의 길이, 폭, 분기 패턴을 파라미터로 제어할 수 있다.
- **성공 조건**:
  - 크랙 생성 요청 시 지정된 표면 위치에 크랙 비주얼이 생성된다
  - 크랙 파라미터(길이, 폭, 심각도)를 변경하면 시각적 결과가 달라진다
  - MVP에서 최소 3개의 크랙이 서로 다른 위치에 배치된다
  - 크랙이 구조물 표면에 자연스럽게 부착되어 표시된다 (공중에 떠있지 않음)
- **실패 조건**:
  - 크랙 생성 시 에러가 발생한다
  - 크랙이 구조물 표면이 아닌 빈 공간에 렌더링된다
  - 동일 파라미터로 생성해도 매번 완전히 동일한 결과만 나온다 (절차적 생성의 의미 없음)
- **예외 처리**:
  - 유효하지 않은 표면 좌표가 입력 -> 가장 가까운 유효 표면 위치로 보정하고 경고 로그 출력
- **대안 동작**: 절차적 생성 실패 시 미리 준비된 기본 크랙 텍스처를 표면에 표시
- **비고**: 난이도 파라미터(SPEC-HAZ-002)와 연동.

### SPEC-ENV-003: 추가 현장 유형 확장 구조

- **출처**: 요구사항 10절 (터널, 도로, 교량 등)
- **설명**: 건물 골조 이외의 현장 유형(터널, 도로, 교량 등)을 추가할 수 있는 확장 구조를 정의한다. 새 현장 유형 추가 시 기존 시스템(위험 요소, 시나리오 관리, 데이터 수집)이 그대로 동작해야 한다.
- **성공 조건**:
  - 새 현장 유형을 추가하고 시나리오 설정 파일에서 현장 유형을 지정하면, 해당 현장이 로드된다
  - 새 현장에서 기존 위험 요소 시스템(SPEC-HAZ-001)이 정상 동작한다
  - 새 현장 추가 시 기존 핵심 코드 수정 없이 현장 정의와 설정만으로 가능하다
- **실패 조건**:
  - 새 현장 추가 시 기존 핵심 코드 수정이 필요하다
- **예외 처리**:
  - 존재하지 않는 현장 유형 지정 -> 기본 현장(건물 골조)으로 폴백하고 경고 로그 출력
- **대안 동작**: 없음
- **비고**: MVP 구현 시 현장 유형을 하드코딩하지 않고 추상화하여 설계해야 한다.

### SPEC-HAZ-001: 위험 요소 기본 시스템 (배치 및 상태 관리)

- **출처**: 요구사항 2.2절 (위험 요소 목록), 9절 MVP (크랙 3개 배치, 발견 처리)
- **설명**: 위험 요소를 씬에 배치하고 상태(미발견/발견)를 관리하는 기본 시스템을 구현한다. 위험 요소는 탐지 가능 영역을 가지며, 플레이어가 마킹(SPEC-INP-002)하면 발견 상태로 전환된다.
- **성공 조건**:
  - 위험 요소가 탐지 가능 영역을 가진다
  - 위험 요소의 초기 상태가 "미발견(undiscovered)"이다
  - 마킹 이벤트 수신 시 해당 위험 요소의 상태가 "발견(discovered)"으로 변경된다
  - 발견 시 시각적 피드백(하이라이트 또는 아이콘)이 표시된다
  - 이미 발견된 위험 요소를 재마킹해도 에러가 발생하지 않으며 중복 기록되지 않는다
  - 현재 씬의 모든 위험 요소 목록을 조회할 수 있다
- **실패 조건**:
  - 마킹해도 상태가 변경되지 않는다
  - 위험 요소 탐지 영역이 비주얼과 불일치한다 (보이지 않는 곳을 마킹해도 발견됨)
- **예외 처리**:
  - 위험 요소가 0개인 씬 -> 경고 로그 출력 후 정상 진행 (빈 시나리오)
- **대안 동작**: 없음
- **비고**: 위험 요소 종류별 확장 가능한 구조로 설계하여 SPEC-HAZ-003 확장에 대비.

### SPEC-HAZ-002: 위험 요소 난이도 파라미터

- **출처**: 요구사항 2.3절 (난이도 가변, 명확한 크랙 ~ 미세 크랙)
- **설명**: 위험 요소의 난이도를 파라미터로 조절할 수 있도록 한다. 난이도는 크기, 명확도(투명도/색상 대비), 위치 접근성 등을 제어한다.
- **성공 조건**:
  - 위험 요소에 난이도 파라미터(0.0~1.0 범위)를 설정할 수 있다
  - 난이도 값 변경 시 위험 요소의 시각적 표현이 변경된다 (예: 0.1 = 큰 크랙, 0.9 = 미세 크랙)
  - 시나리오 설정 파일(SPEC-SCN-001)에서 개별 위험 요소의 난이도를 지정할 수 있다
  - 난이도 최솟값과 최댓값에서 시각적으로 구분 가능한 차이가 있다
- **실패 조건**:
  - 난이도를 변경해도 시각적 차이가 없다
  - 범위 밖 값(음수, 1.0 초과) 입력 시 에러가 발생하거나 비정상 렌더링된다
- **예외 처리**:
  - 범위 밖 값 입력 -> 0.0~1.0 범위로 clamp 처리하고 경고 로그 출력
- **대안 동작**: 없음
- **비고**: 연구 조건에 따라 난이도 체계가 변경될 수 있으므로 파라미터 확장이 용이한 구조로 설계.

### SPEC-HAZ-003: 위험 요소 종류 확장 (부식, 누수 등)

- **출처**: 요구사항 2.2절 (7종 위험 요소), 10절 (위험 요소 종류 추가)
- **설명**: 크랙 이외의 위험 요소 종류(철근 부식, 누수, 지반 침하, 붕괴 조짐, 가시설 변형, 안전 장비 미비)를 추가 구현한다.
- **성공 조건**:
  - 각 위험 요소 종류가 공통 인터페이스를 구현한 고유 유형으로 존재한다
  - 새 위험 요소 종류 추가 시 공통 인터페이스를 구현하면 기존 시스템에서 자동 인식된다
  - 시나리오 설정 파일에서 종류별 배치가 가능하다
  - 각 종류별로 시각적으로 구분 가능한 고유 표현이 있다
- **실패 조건**:
  - 새 위험 요소 종류 추가 시 기존 핵심 코드 수정이 필요하다
  - 기존 크랙 시스템에 영향을 주어 회귀 버그가 발생한다
- **예외 처리**:
  - 알 수 없는 위험 요소 종류가 설정 파일에 지정 -> 경고 로그 출력하고 해당 항목 건너뜀
- **대안 동작**: 없음
- **비고**: 위험 요소 공통 인터페이스 구조는 SPEC-HAZ-001에서 사전 설계. 각 종류별 절차적 생성 로직은 별도 개발.

### SPEC-INP-001: 조이스틱 기반 이동

- **출처**: 요구사항 3절 (컨트롤러 조이스틱으로 이동), 9절 MVP
- **설명**: VR 컨트롤러의 조이스틱 입력으로 플레이어를 3D 환경 내에서 이동시킨다. 왼쪽 조이스틱으로 전후좌우 이동, 오른쪽 조이스틱으로 회전(snap turn)을 지원한다.
- **성공 조건**:
  - 왼쪽 조이스틱 전방 입력 시 카메라가 바라보는 방향으로 이동한다
  - 왼쪽 조이스틱 좌우 입력 시 횡이동(strafe)한다
  - 오른쪽 조이스틱 좌우 입력 시 시점이 회전한다 (snap turn, 30도 또는 45도 단위)
  - 이동 속도가 설정 가능하다 (기본값 3.0 m/s)
  - 벽이나 구조물에 부딪히면 이동이 차단된다 (충돌 판정 적용)
  - 바닥 위에서만 이동하며 공중에 뜨지 않는다
- **실패 조건**:
  - 조이스틱 입력에 캐릭터가 반응하지 않는다
  - 이동 시 벽을 관통한다
  - VR 멀미를 유발하는 급격한 카메라 흔들림이 발생한다
- **예외 처리**:
  - 컨트롤러 연결이 끊어짐 -> "컨트롤러를 다시 연결하세요" 메시지 표시, 재연결 시 자동 복구
- **대안 동작**: VR 컨트롤러 미사용 시 키보드+마우스로 이동 (SPEC-VR-002 데스크톱 모드)
- **비고**: VR 멀미 방지를 위해 snap turn 사용 (smooth turn은 옵션으로 제공 가능).

### SPEC-INP-002: 컨트롤러 버튼 마킹

- **출처**: 요구사항 3절 (대상을 바라보고 컨트롤러 버튼으로 마킹), 9절 MVP
- **설명**: 플레이어가 위험 요소를 바라본 상태에서 컨트롤러 버튼을 누르면 마킹(발견 처리)한다. 화면 중심에서 전방으로 광선 판정(ray casting)을 수행하여 대상을 판별한다.
- **성공 조건**:
  - 트리거 버튼 입력 시 카메라 중심에서 전방으로 광선 판정이 수행된다
  - 광선이 위험 요소의 탐지 영역에 적중하면 해당 위험 요소가 발견 처리된다 (SPEC-HAZ-001 연동)
  - 광선이 위험 요소에 적중하지 않으면 오탐(false positive)으로 기록된다 (SPEC-DAT-004 연동)
  - 마킹 성공 시 시각적 피드백(마커 표시)과 청각적 피드백(효과음)이 제공된다
  - 마킹 실패(빈 공간) 시에도 다른 피드백이 제공된다
  - 광선의 최대 탐지 거리가 설정 가능하다 (기본값 50m)
- **실패 조건**:
  - 버튼을 눌러도 마킹이 동작하지 않는다
  - 위험 요소를 정면으로 바라보고 마킹했는데 인식되지 않는다
  - 위험 요소가 아닌 곳을 마킹했는데 발견으로 처리된다
- **예외 처리**:
  - 광선이 아무것도 적중하지 않음 (허공) -> 오탐으로 기록하되 마킹 위치는 광선 끝점으로 기록
- **대안 동작**: 없음
- **비고**:
  - 광선 시각화(레이저 포인터 또는 크로스헤어)는 연구 목적에 따라 on/off 가능하게 설계.
  - 시각 마커(HazardMarkPlaced): 마킹 적중 위치에 영구 시각 마커를 spawn한다. 메타데이터(Vector3 위치 + Unix epoch ms timestamp + 카테고리)는 세션 채점(SPEC-DAT-002)에서 실제 hazard와 매칭하는 입력으로 사용된다. 시그널 명: `hazard_mark_placed` (대응 fl: HazardMarkPlaced).

### SPEC-INP-003: 화면 중심 기반 시선 추적

- **출처**: 요구사항 3절 (화면 중심 기반 시선 추적)
- **설명**: 사용자의 시선 방향을 화면(카메라) 중심 기준으로 추정한다. 매 샘플링 주기마다 카메라의 전방 벡터를 기록한다.
- **성공 조건**:
  - 매 샘플링 주기마다 카메라의 전방 방향 벡터가 기록된다
  - 샘플링 주기가 설정 가능하다 (기본값 100ms)
  - 기록에 타임스탬프가 포함된다
  - 기록된 시선 데이터를 후처리 분석에 활용할 수 있는 구조화된 형식으로 저장된다
- **실패 조건**:
  - 시선 데이터가 기록되지 않는다
  - 타임스탬프가 누락되거나 순서가 역전된다
- **예외 처리**:
  - 카메라가 준비되지 않음 -> 에러 로그 출력 후 시선 추적 비활성화
- **대안 동작**: 없음
- **비고**: HMD 내장 eye tracker 활용은 향후 확장(요구사항 10절). 현재는 화면 중심 기반으로만 구현.

### SPEC-SCN-001: 시나리오 설정 파일

- **출처**: 요구사항 6절 (코드 수정 없이 위험 요소 배치 변경, 설정 파일)
- **설명**: 연구자가 코드 수정 없이 시나리오를 구성할 수 있는 외부 설정 파일을 제공한다. 위험 요소의 종류, 위치, 난이도 등을 외부 파일로 정의한다.
- **성공 조건**:
  - 구조화된 설정 파일을 읽어 위험 요소를 배치할 수 있다
  - 설정 파일에 위험 요소 종류, 위치, 난이도 필드가 포함된다
  - 설정 파일에 세션 시간 제한, 현장 유형 필드가 포함된다
  - 유효한 설정 파일 로드 시 씬에 해당 위험 요소가 정확히 배치된다
  - 설정 파일 변경 후 재실행하면 변경된 배치가 반영된다
  - 설정 파일 스키마 검증이 수행된다
- **실패 조건**:
  - 설정 파일을 읽지 못하거나 파싱 에러가 발생하되 사용자에게 알리지 않는다
  - 설정 파일의 위치와 실제 배치 위치가 불일치한다
- **예외 처리**:
  - 설정 파일이 존재하지 않음 -> 기본 시나리오(MVP 크랙 3개)로 실행하고 로그 출력
  - 설정 파일 파싱 실패 -> 구체적 에러 위치를 포함한 에러 메시지 출력 후 기본 시나리오로 실행
  - 설정 파일 내 좌표가 씬 범위 밖 -> 해당 위험 요소 건너뛰고 경고 출력
- **대안 동작**: 설정 파일 없이도 기본값으로 실행 가능
- **비고**: 예시 설정 파일 템플릿을 함께 제공한다.

### SPEC-SCN-002: 위험 요소 랜덤 배치

- **출처**: 요구사항 6절 (위험 요소 랜덤 배치 기능 지원)
- **설명**: 위험 요소를 구조물 표면 위 유효한 위치에 랜덤으로 배치하는 기능을 제공한다. 시나리오 설정 파일에서 랜덤 배치 모드를 활성화할 수 있다.
- **성공 조건**:
  - 시나리오 설정에서 랜덤 배치 모드를 활성화하면 위험 요소가 랜덤 위치에 배치된다
  - 랜덤 배치된 위험 요소가 모두 구조물 표면 위에 위치한다 (공중/지하 배치 없음)
  - 랜덤 시드를 지정하면 동일한 배치가 재현된다
  - 위험 요소 간 최소 간격이 유지된다 (겹침 방지)
  - 랜덤 배치 시 위험 요소 개수와 종류를 설정에서 지정할 수 있다
- **실패 조건**:
  - 랜덤 배치된 위험 요소가 도달 불가능한 위치에 있다
  - 위험 요소가 서로 겹친다
  - 동일 시드에서 다른 배치가 생성된다
- **예외 처리**:
  - 유효한 배치 위치가 부족 -> 가능한 만큼만 배치하고 실패한 개수를 경고 로그로 출력
- **대안 동작**: 랜덤 배치 실패 시 기본 고정 배치로 폴백
- **비고**: 연구 재현성을 위해 시드 기반 랜덤 필수.

### SPEC-DAT-001: 세션 결과 로컬 파일 저장

- **출처**: 요구사항 5.1절 (로컬 CSV/JSON), 5.2절 (기록 항목), 9절 MVP
- **설명**: 한 세션의 결과 데이터를 로컬 파일(CSV 및 JSON)로 저장한다. 피험자 정보, 세션 설정, 위험 요소별 발견 결과를 포함한다.
- **성공 조건**:
  - 세션 종료 시 사용자 데이터 경로에 결과 파일이 생성된다
  - 파일명에 피험자 ID와 타임스탬프가 포함된다
  - 결과 파일에 다음 정보가 포함된다:
    - 피험자 ID
    - 세션 시작 시각 (표준 시간 형식)
    - 사용된 시나리오 설정
    - 제한 시간
    - 위험 요소별 발견 여부, 발견 시각, 마킹 시 사용자 위치, 마킹 시 시선 방향
    - 발견율
    - 총 소요 시간
  - 스프레드시트 호환 형식의 결과 파일이 별도로 생성된다
  - 기존 파일이 있으면 덮어쓰지 않고 새 파일을 생성한다
- **실패 조건**:
  - 세션 종료 시 파일이 생성되지 않는다
  - 파일이 생성되었으나 내용이 비어있거나 파싱 불가능하다
  - 필수 정보가 누락된다
- **예외 처리**:
  - 파일 쓰기 권한 없음 -> 에러 메시지 표시, 대체 경로로 시도
  - 디스크 공간 부족 -> 에러 메시지 표시, 최소 데이터(발견율, 반응시간)만 콘솔에 출력
- **대안 동작**: 파일 저장 실패 시 콘솔에 결과 요약을 출력
- **비고**: 상세 데이터 형식과 요약 분석 형식 두 가지 모두 생성.

### SPEC-DAT-002: 발견율 및 반응 시간 산출

- **출처**: 요구사항 4.1절 (발견율, 반응 시간), 9절 MVP
- **설명**: 세션 중 사용자의 위험 요소 발견율과 개별 위험 요소에 대한 반응 시간을 실시간으로 산출한다.
- **성공 조건**:
  - 발견율 = (발견한 위험 요소 수 / 전체 위험 요소 수) * 100 으로 계산된다
  - 반응 시간 = (위험 요소 발견 시각 - 세션 시작 시각)으로 개별 위험 요소마다 기록된다
  - 세션 종료 시 전체 발견율과 각 위험 요소별 반응 시간이 결과 데이터에 포함된다
  - 발견율은 소수점 1자리까지, 반응 시간은 밀리초 단위로 기록된다
  - 미발견 위험 요소의 반응 시간은 미측정 값으로 기록된다
- **실패 조건**:
  - 발견율 계산 결과가 0~100 범위를 벗어난다
  - 반응 시간이 음수이다
  - 전체 위험 요소 수와 발견+미발견 수의 합이 불일치한다
- **예외 처리**:
  - 위험 요소가 0개인 세션 -> 발견율을 0%로 기록하고 경고 로그 출력
- **대안 동작**: 없음
- **비고**:
  - 실시간 산출 결과는 세션 중에는 내부적으로만 유지하고, 세션 종료 후 SPEC-DAT-001을 통해 저장. 실시간 UI 표시는 연구 목적에 따라 선택적.
  - 사후 채점(SessionScorer): 실제 hazard 위치와 사용자 마킹(HazardMarkPlaced)을 거리 임계(2m)로 매칭하여 precision/recall/F1, true_positives/false_positives/false_negatives, 평균 식별 시간을 산출한다. GDScript `SessionScorer.score()` 및 Python `tools/score_session.py`에서 동일 알고리즘 제공 (오프라인 분석 가능).

### SPEC-DAT-003: 뇌파(EEG) 동기화용 타임스탬프 로깅

- **출처**: 요구사항 4.2절 (타임스탬프 기반 후처리 동기화), 5.3절
- **설명**: VR 시뮬레이션의 주요 이벤트에 고정밀 타임스탬프를 기록하여, 별도 기록되는 EEG 데이터와 후처리 동기화를 가능하게 한다.
- **성공 조건**:
  - 다음 이벤트에 타임스탬프가 기록된다: 세션 시작, 세션 종료, 위험 요소 발견, 마킹 시도(성공/실패), 이동 시작/정지
  - 타임스탬프는 Unix epoch 기준 밀리초 단위이다
  - 타임스탬프 이벤트 로그가 별도 파일로 저장된다
  - 이벤트 로그에 이벤트 타입, 타임스탬프, 관련 데이터(위치, 대상 등)가 포함된다
  - 세션 시작 시 절대 시간(시스템 시각)이 기록되어 외부 기기와 동기화 기준점을 제공한다
- **실패 조건**:
  - 타임스탬프가 누락된 이벤트가 존재한다
  - 타임스탬프의 정밀도가 100ms보다 나쁘다
  - 이벤트 순서와 타임스탬프 순서가 불일치한다
- **예외 처리**:
  - 시스템 시계 이상 -> 상대 시간(세션 시작 이후 경과 시간)으로 폴백
- **대안 동작**: 없음
- **비고**: EEG 기기 미정이므로 범용적 형식으로 기록.

---

### SPEC-DAT-004: 사용자 행동 로깅 (이동 경로, 시선, 오탐)

- **출처**: 요구사항 5.2절 (이동 경로, 시선 방향, 오탐 기록)
- **설명**: 세션 중 사용자의 이동 경로, 시선 방향, 오탐(위험 요소가 아닌 곳 마킹) 기록을 수집한다.
- **성공 조건**:
  - 이동 경로가 설정된 샘플링 주기(기본 200ms)로 기록된다 (타임스탬프, 3D 좌표)
  - 시선 방향이 설정된 샘플링 주기(기본 100ms)로 기록된다 (타임스탬프, 방향 벡터)
  - 오탐 시 타임스탬프, 마킹 위치, 마킹 시 시선 방향이 기록된다
  - 모든 행동 데이터가 별도 파일로 저장된다
  - 대용량 데이터(장시간 세션)에서도 프레임 드롭 없이 기록된다
- **실패 조건**:
  - 행동 데이터 기록으로 인해 프레임레이트가 10% 이상 저하된다
  - 데이터 파일이 손상되거나 불완전하다
- **예외 처리**:
  - 저장 버퍼 오버플로우 -> 배치 쓰기로 전환하여 IO 부하 분산
- **대안 동작**: 성능 문제 발생 시 샘플링 주기를 자동으로 2배 늘림
- **비고**: 메모리 버퍼에 누적 후 주기적으로 파일에 flush하는 방식으로 IO 부하 최소화.

### SPEC-DAT-005: 뇌파 기기 실시간 연동 (LSL)

- **출처**: 요구사항 10절 (뇌파 기기 실시간 연동, LSL)
- **설명**: Lab Streaming Layer(LSL) 프로토콜을 통해 뇌파 기기와 실시간 데이터 스트리밍을 지원한다.
- **성공 조건**:
  - LSL 아울렛(outlet)을 생성하여 VR 이벤트를 실시간 스트리밍할 수 있다
  - LSL 인렛(inlet)을 통해 외부 EEG 데이터를 수신할 수 있다
  - LSL 연결 상태를 모니터링하고 UI에 표시한다
- **실패 조건**:
  - LSL 라이브러리 로드 실패 시 VR 시뮬레이션 전체가 영향받는다
- **예외 처리**:
  - LSL 라이브러리 미설치 -> LSL 기능 비활성화, SPEC-DAT-003 타임스탬프 방식으로 폴백
  - LSL 연결 끊김 -> 자동 재연결 시도, 실패 시 타임스탬프 방식으로 폴백하고 경고 표시
- **대안 동작**: SPEC-DAT-003 (타임스탬프 기반 후처리 동기화)
- **비고**: 뇌파 기기 미정이므로 범용 LSL 프로토콜 지원.

### SPEC-SES-001: 피험자 정보 입력 화면

- **출처**: 요구사항 7절 (피험자 정보 입력), 5.2절 (피험자 정보 — ID, 경력 등)
- **설명**: 시뮬레이션 시작 전 피험자 정보를 입력받는 UI 화면을 제공한다.
- **성공 조건**:
  - 피험자 ID 입력 필드가 존재하며 필수값이다
  - 경력 정보(연차 또는 카테고리) 입력 필드가 존재한다
  - 시작 버튼을 누르면 입력된 정보가 세션 데이터에 저장되고 시뮬레이션이 시작된다
  - 피험자 ID가 비어있으면 시작 버튼이 비활성화되거나 경고가 표시된다
  - VR 환경에서도 조작 가능한 UI이다 (컨트롤러로 입력 가능)
- **실패 조건**:
  - 피험자 정보 없이 시뮬레이션이 시작된다
  - 입력한 정보가 결과 파일에 반영되지 않는다
- **예외 처리**:
  - 중복 피험자 ID -> 경고 표시 후 사용자 확인 시 진행 (덮어쓰기 아님, 별도 세션으로 저장)
- **대안 동작**: 데스크톱 모드에서는 일반 키보드로 입력
- **비고**: MVP에서는 최소 필드(ID, 경력 선택)만 제공. 추가 필드는 연구 필요에 따라 확장.

### SPEC-SES-002: 세션 타이머 (시간 제한)

- **출처**: 요구사항 7절 (시간 제한 있음)
- **설명**: 시뮬레이션 세션에 시간 제한을 적용하고, 제한 시간 도달 시 세션을 자동 종료한다.
- **성공 조건**:
  - 시나리오 설정(SPEC-SCN-001) 또는 기본값으로 시간 제한(초 단위)이 설정된다
  - 타이머가 세션 시작과 동시에 카운트다운을 시작한다
  - 남은 시간이 HUD에 표시된다 (분:초 형식)
  - 시간이 0에 도달하면 세션이 자동 종료되고 결과가 저장된다 (SPEC-DAT-001)
  - 제한 시간 전에 모든 위험 요소를 발견하면 조기 종료 옵션이 제공된다
  - 기본 시간 제한은 300초(5분)이다
- **실패 조건**:
  - 시간 제한에 도달해도 세션이 종료되지 않는다
  - 타이머 표시와 실제 경과 시간이 1초 이상 차이난다
- **예외 처리**:
  - 시간 제한이 0 또는 음수로 설정 -> 기본값(300초)으로 대체하고 경고 로그 출력
- **대안 동작**: 없음
- **비고**: 시간 제한은 연구 조건에 따라 변경 가능해야 한다. HUD 타이머 표시 여부도 설정 가능하게 (연구자가 피험자에게 시간 압박을 주고 싶지 않을 수 있음).

### SPEC-SES-003: 세션 흐름 제어 (시작-진행-종료)

- **출처**: 요구사항 7절 (기본 흐름: 정보 입력 -> 시뮬레이션 -> 결과 저장)
- **설명**: 세션의 전체 생명주기를 관리하는 상태 머신을 구현한다. 각 단계(초기화, 정보입력, 시뮬레이션, 결과, 종료) 간 전환을 제어한다.
- **성공 조건**:
  - 세션 상태가 다음 순서로 전환된다: 초기화 -> 정보입력 -> 진행 -> 결과 -> 종료
  - 각 상태 전환 시 이벤트가 발생하여 다른 시스템이 반응할 수 있다
  - 초기화: 씬 로드, 시나리오 설정 적용
  - 정보입력: 피험자 정보 입력 대기 (SPEC-SES-001)
  - 진행: 시뮬레이션 진행, 타이머 동작 (SPEC-SES-002)
  - 결과: 결과 표시 및 저장 (SPEC-DAT-001)
  - 종료: 다음 세션 또는 종료 선택
  - 비정상 종료(앱 크래시 제외) 시에도 수집된 데이터가 저장된다
- **실패 조건**:
  - 상태 전환이 올바른 순서를 따르지 않는다 (예: 정보입력에서 바로 종료)
  - 특정 상태에서 무한 대기가 발생한다
- **예외 처리**:
  - 시뮬레이션 중 사용자가 강제 종료 요청 -> 현재까지의 데이터를 저장 후 결과 상태로 전환
- **대안 동작**: 없음
- **비고**: 상태 패턴(State Pattern) 기반 구현 권장.
